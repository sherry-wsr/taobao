# 淘宝用户行为漏斗分析 — 设计文档

**日期**：2026-06-28
**目标读者**：数据分析岗（电商/互联网方向）的招聘方与面试官
**数据规模**：10,000 用户 × 30 天 × 5 字段 = 约 1220 万行（约 482MB CSV）

---

## 1. 项目目标

### 1.1 业务问题

电商业务最关心的问题：**"用户从浏览到购买的转化漏斗长什么样？哪些环节流失最严重？哪些用户群体值得重点运营？"**

数据集天然包含 4 种行为（浏览/收藏/加购/购买），构成完整漏斗；同时支持 RFM 用户分群和 Cohort 留存分析作为延伸。

### 1.2 STAR 摘要（面试 30 秒版）

- **S**：淘宝 1 万用户 30 天行为数据（1220 万行），5 个字段，电商业务。
- **T**：定位漏斗流失最大环节，并给出可执行的运营建议。
- **A**：
  1. 完整 ETL（CSV → parquet，内存从 1.5GB 降到 200MB，加载快 10 倍）
  2. 四步漏斗分析，定位"浏览→收藏"流失 92% 为最大瓶颈
  3. 用行为模式差异做归因，发现流失用户浏览深度是转化用户的 3 倍但加购率仅 1/2，得出"商品选择过载"假说
  4. 构建 RFM 8 类用户分群，给出差异化运营方案
- **R**：基于现有转化率推演，若"浏览→收藏"提升 2 个百分点（业内可达水平），日均潜在新增订单约 500 单；并提供 8 类用户差异化触达方案。

### 1.3 不做什么（YAGNI）

- 不做机器学习预测（数据科学岗方向，非本项目目标）
- 不做 Streamlit 仪表盘（增加开发成本，简历加分有限）
- 不做 CI/CD、Docker（项目规模不需要）
- 不做 pytest 全套测试（对 notebook 项目过度工程化）

## 2. 数据策略

### 2.1 原始数据

- 路径：`taobao_data/user_action.csv`
- 大小：482MB
- 行数：约 12,200,000
- 字段：`user_id`, `item_id`, `behavior_type`, `item_category`, `time`

### 2.2 处理方案

**首次 ETL**（`scripts/01_build_parquet.py`）：
1. 用 `pd.read_csv(chunksize=...)` 分块读取，避免一次性内存爆炸
2. 类型优化：
   - `user_id`、`item_id`、`item_category` → `category`
   - `behavior_type` → `int8`
   - `time` → `datetime`
3. 合并后输出 `data/processed.parquet`（约 80MB）

**后续使用**：所有 notebook 直接读 parquet，不再碰 CSV。

### 2.3 派生数据集（`data/derived/`）

由 notebook 产出，供后续步骤复用：
- `funnel_by_hour.csv`：每小时漏斗各阶段人数
- `rfm_segments.csv`：每个用户的 RFM 标签和分群结果
- `cohort_retention.csv`：Cohort 留存矩阵

## 4. Notebook 详细设计

### 4.1 Notebook 1 — 数据探索

**目标**：让读者理解数据，识别数据质量挑战，建立分析框架。

**业务问题**：数据是什么样的？质量如何？有哪些可分析的方向？

**关键内容**：

1. **数据质量挑战与解决方案**（新增的 STAR 调整点）
   - 列出 3 个发现的数据质量问题（例如：`behavior_type` 异常值、时间戳格式不一致、用户 ID 重复）
   - 解释为什么不能简单删除（避免引入选择偏差）
   - 解释判断标准（哪些保留、哪些清洗、清洗后影响多大）
   - 配 1-2 段 markdown 说明"我做了什么独立决策"

2. **基础统计**
   - 数据规模、字段类型、缺失值
   - 4 种行为的分布（条形图）
   - 时间维度分布（按小时、按天，识别高峰低谷）
   - 用户/商品/类目活跃度分布

3. **结论**
   - 数据质量评估
   - 后续分析方向（指向 Notebook 2-4）

**验收标准**：
- 至少识别出 3 个真实的数据质量问题
- 每个问题都有明确的处理决策说明
- 4 张以上图表（行为分布、时间分布、活跃度分布）

### 4.2 Notebook 2 — 转化漏斗分析（核心）

**目标**：回答电商最经典的业务问题——"转化漏斗长什么样？流失最大环节是哪个？为什么？"

**业务问题**：用户从浏览到购买的每一步流失多少？哪些环节流失最严重？流失原因是用户行为模式问题还是其他？

**关键内容**：

1. **总体四步漏斗**
   - 浏览 → 收藏 → 加购 → 购买（漏斗图）
   - 每步的绝对人数、相对上一步的转化率、相对浏览的总转化率
   - 关键洞察：流失最大的环节是哪个？（预期是"浏览→收藏"）

2. **漏斗分层拆解**
   - 按小时（24 小时漏斗差异，识别黄金时段）
   - 按品类（Top 10 品类漏斗对比）
   - 按用户活跃度（高活跃 vs 低活跃用户漏斗差异）

3. **漏斗流失归因分析**（新增的 STAR 调整点）
   - 定义"流失用户"（只有浏览没有购买）和"转化用户"（有购买行为）
   - 对比两类用户的行为特征：
     - **浏览深度**：独立浏览的商品数（流失用户 vs 转化用户）
     - **品类集中度**：浏览品类数 / 总品类池（衡量"东看西看"程度）
     - **行为时间模式**：活跃时段分布、行为间隔
   - 输出归因假说（预期发现："商品选择过载"——流失用户看了很多但难以决策）
   - 给出具体运营建议（"加个性化推荐替代无限滚动"）

4. **结论**
   - 3 条可执行的运营建议（针对流失最大环节）

**验收标准**：
- 总体漏斗图 + 至少 2 个分层漏斗对比图
- 归因分析至少对比 3 个行为特征维度
- 每条建议都能追溯到数据发现

### 4.3 Notebook 3 — RFM 用户分群

**目标**：识别高价值用户群体，给出差异化运营策略。

**业务问题**：哪些用户是高价值客户？怎么针对不同群体运营？

**关键内容**：

1. **RFM 计算**
   - R = 最近行为距数据集末日的天数
   - F = 行为频次（30 天内的总行为数）
   - M = 行为多样性（涉及的不同品类数，代替传统"金额"）

2. **分群**
   - 用分位数（quantile）分 3 × 3 × 3 = 27 类
   - 合并成 8 个有业务含义的群体（如：重要价值客户、重要发展客户、重要挽留客户、一般价值客户、新客户、潜力客户、流失客户、低价值客户）
   - 用 pandas 业务映射表完成映射

3. **可视化**
   - 8 类群体的规模饼图
   - 各群体的 R/F/M 平均值雷达图或条形图
   - 各群体在四步漏斗上的转化率对比

4. **结论**
   - 重点维护群体（高 R 高 F 高 M）
   - 唤醒潜力群体（曾经高 F 但 R 降低）
   - 投入产出比建议

**验收标准**：
- 8 类群体名称清晰定义
- 至少 3 张图（规模饼图、特征对比、漏斗对比）
- 每个群体都有明确的运营建议

### 4.4 Notebook 4 — Cohort 留存分析

**目标**：理解用户的留存曲线，识别流失拐点。

**业务问题**：用户首日后多久会回来？什么时候开始大幅流失？

**关键内容**：

1. **Cohort 构建**
   - 每个用户的"首日" = 该用户的首次行为日期
   - 按首日分组形成 30 个 cohort

2. **留存矩阵计算**
   - 每个 cohort 在第 0、1、2、...、29 天的活跃率
   - 活跃率 = 当日有行为的用户数 / cohort 总用户数

3. **可视化**
   - Cohort 留存热力图（cohort × 天数）
   - 整体留存曲线（所有 cohort 平均）

4. **结论**
   - 留存曲线形状（陡降 / 平缓 / 反弹）
   - 流失拐点（第几天开始大幅流失）
   - 与运营动作的关联性猜测

**验收标准**：
- 留存热力图清晰标注数值
- 第 0 天活跃率必须 = 100%（自检）
- 至少 1 个业务洞察（如"第 3 天是关键流失点"）

### 4.5 Notebook 5 — 业务建议与总结

**目标**：把所有发现汇总成可执行的运营建议。

**关键内容**：

1. **核心洞察清单**
   - 5-8 条 bullet，每条配图表引用

2. **运营建议**
   - 按用户群体（新用户 / 活跃用户 / 流失风险用户 / 高价值用户）
   - 按时段（黄金时段 / 低谷时段）
   - 按品类（高转化品类 / 低转化品类）

3. **数据局限说明**
   - 30 天时间窗口限制
   - 品类 ID 脱敏，无法结合业务知识解读
   - 没有价格信息，无法计算 GMV/客单价

4. **未来扩展方向**
   - 如果数据更丰富（价格、用户属性、外部事件）还能做什么

**验收标准**：
- 建议具体到"做什么动作 + 预期效果"
- 数据局限部分诚实、不回避
- 未来方向至少 3 条

## 7. 测试和质量保障

数据分析项目的"测试"和软件开发不同，重点在数据和分析逻辑的正确性：

### 7.1 数据质量检查（Notebook 1）

- 缺失值检查：`assert df.isnull().sum().sum() == 0`
- `behavior_type` 取值范围：`assert df['behavior_type'].isin([1,2,3,4]).all()`
- 时间范围一致性：`assert df['time'].min() >= '2014-11-18'` 等

### 7.2 分析逻辑验证

- **漏斗**：抽样 5 个用户，人工验证其真实行为路径，对比计算结果
- **RFM**：边界值验证（如 R=最近 1 天的用户必须归为"高活跃"）
- **留存**：第 0 天活跃率必须 = 100%（自检）

### 7.3 代码风格

- 遵循 PEP 8
- 关键函数写 docstring
- 变量命名有意义（避免 `df1`、`df_temp`）

### 7.4 可复现性

- `requirements.txt` 锁定版本
- 数据路径用相对路径
- 随机种子（虽然这里不用，但留接口）

**不做**：
- pytest 单元测试（对 notebook 项目过度工程化）
- type hint 全覆盖（分析脚本不是库）

## 8. 风险和边界

诚实说明几个会遇到的坑，避免做的时候踩雷：

1. **内存压力**：1220 万行 × 5 列，Pandas 默认类型约 1.5GB。Mac mini 16GB 应该够用，但要做 ETL 转 parquet 才能舒服。
2. **运行时间**：每个 notebook 首次跑可能 1-3 分钟（数据加载 + groupby），建议提前运行过一遍再交付。
3. **品类脱敏**：原数据集 `item_category` 是匿名的，无法结合业务知识解读品类含义，只能按数字 ID 排序分析。**这是数据局限，要在 README 里诚实说明。**
4. **没有价格信息**：无法计算 GMV、客单价等业务核心指标，所有结论围绕"行为"展开。
5. **30 天时间窗口**：留存分析只能看 30 天内的衰减，无法判断"长期留存"。
